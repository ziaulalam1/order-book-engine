# LOBSTER Replay + Comparison Tool

Streams LOBSTER-format event CSVs through `OrderBook` (from `../engine.js`) and verifies engine state against the LOBSTER reconstructed L2 at every checkpoint.

LOBSTER (`lobsterdata.com`) is a Berlin-based academic project that reconstructs limit-order books from raw NASDAQ Historical TotalView-ITCH data. The samples on `lobsterdata.com/info/DataSamples.php` are derived from the official NASDAQ TotalView-ITCH sample for academic use. Each sample is two paired CSVs: one event stream (`*_message.csv`) and the reconstructed orderbook state at every event (`*_orderbook.csv`). The orderbook CSV is what makes this useful as a verification reference: if our engine reproduces LOBSTER's L2 at every event, the engine is correct against real exchange data without us needing direct ITCH access.

## Files in this directory

| File | Purpose |
|------|---------|
| `replay_lobster.js` | CLI replay tool. Streams events into `OrderBook`, checkpoints engine snapshot vs LOBSTER orderbook CSV. |
| `synth.js` | Deterministic synthetic CSV generator mirroring LOBSTER schema. Used as the verification reference when real samples are not reachable. |
| `tests/replay.test.js` | Three-scenario test suite. Hand-crafted 6-event walkthrough, 200-event synth roundtrip with per-event checkpoint, 5000-event synth at 500-event checkpoint cadence. |
| `data/` | Sample CSVs (gitignored). User drops real LOBSTER samples here, or `synth.js` writes here. |
| `.gitignore` | Excludes `data/*` and tmp artifacts. |

## LOBSTER schema

Reference: `lobsterdata.com/info/DataStructure.php`.

### `*_message.csv` columns

| Idx | Field | Notes |
|-----|-------|-------|
| 0 | time | seconds after midnight, fractional |
| 1 | type | 1=submission, 2=cancel partial, 3=delete, 4=exec visible, 5=exec hidden, 6=cross trade, 7=halt |
| 2 | order_id | unique per submission; 0 for hidden/halt |
| 3 | size | shares (for type 2 and 4: the cancel/exec delta, not resulting size) |
| 4 | price | dollar price * 10000 (so $50.0001 -> 500001) |
| 5 | direction | -1 sell limit order, +1 buy limit order. For exec events this is the maker side, not the taker. |

### `*_orderbook.csv` columns (`--levels N`)

Alternating: `ask_price_1, ask_size_1, bid_price_1, bid_size_1, ask_price_2, ask_size_2, ...`

Empty levels: `ask_price = 9999999999`, `bid_price = -9999999999`, size = 0.

## Mapping LOBSTER to engine API

`engine.js` exports `OrderBook.submitOrder(side, type, price, qty)` and `cancelOrder(id)`. It does not expose partial-qty mutation. To handle types 2 and 4 without modifying `engine.js` (forbidden by sprint constraint), `replay_lobster.js` walks `book.bids` and `book.asks` directly and mutates the matching maker order in place. This preserves time priority on partial events, which `cancel + resubmit` would not.

| LOBSTER type | Engine action |
|-----|-----|
| 1 | `submitOrder(side, 'LIMIT', price, size)`; map `lobster_id -> engine_id` |
| 2 | walk `book.bids`/`book.asks`, find engine_id, decrement `qty -= size`; remove if zero |
| 3 | `cancelOrder(engine_id)` |
| 4 | locate maker, decrement, push synthetic trade with inferred taker side, update vwap |
| 5 | skip (hidden orders are off-book in LOBSTER's reconstructed L2) |
| 6 | skip (cross trades, rare) |
| 7 | log only, no engine action |

Side derivation: LOBSTER `direction = +1` is a buy limit (engine `BUY`), `-1` is a sell limit (`SELL`). For execution events, the maker side comes from the resting order's stored direction; the taker side is the opposite.

Price scaling: LOBSTER prices are integer (dollar * 10000). `OrderBook` accepts any numeric price, so we keep them as integers throughout to avoid float comparison issues.

## CLI

```sh
node replay_lobster.js <message.csv> <orderbook.csv> \
  [--levels 10] [--checkpoint-every 1000] [--limit N] [--quiet]
```

Output: per-checkpoint pass/fail line, then a summary block with event-type counts, parser errors, unmapped order references, checkpoints checked, mismatches.

Exit code: 0 on zero mismatches, 1 otherwise.

### Synthetic data

```sh
node synth.js data/sample --events 5000 --levels 10 --seed 1
```

Writes `data/sample_message.csv` and `data/sample_orderbook.csv`. The orderbook file is generated by an internal reference implementation that mirrors the engine's expected behavior, so a clean replay of the synthetic message file should produce zero mismatches at any checkpoint cadence.

## Real LOBSTER samples (user step)

Direct programmatic download from `lobsterdata.com` was not available from this environment (the demo zip URLs returned an HTML page instead of a binary archive when probed on 2026-04-30, likely an anti-leech check). To use real samples:

1. Visit `https://lobsterdata.com/info/DataSamples.php` in a browser.
2. Pick a sample (e.g., AMZN level 10 quiet day) and download the zip.
3. Unzip and place `*_message.csv` and `*_orderbook.csv` into `lobster/data/`.
4. Run replay against them:
   ```sh
   node replay_lobster.js data/AMZN_2012-06-21_34200000_57600000_message_10.csv \
                          data/AMZN_2012-06-21_34200000_57600000_orderbook_10.csv \
                          --levels 10 --checkpoint-every 1000
   ```

Sample data files are excluded from version control via `.gitignore`. LOBSTER's terms permit academic-use download; redistribution is restricted, so committing the data files is not appropriate.

## Tests

```sh
node tests/replay.test.js
```

Three scenarios. All pass on synthetic input as of 2026-04-30:
- Hand-crafted 6-event scenario with hand-computed orderbook rows (checkpoints every event, asserts trade and final book state)
- 200-event synth roundtrip (every-event checkpoint, asserts zero mismatches and that all message types appear)
- 5000-event synth at 500-event cadence (broader sweep)

When real LOBSTER samples are dropped in, the same test infrastructure can be pointed at them by adding a fourth scenario; expect tighter assertions because real LOBSTER prices and sizes follow exchange-realistic distributions while synth uses a uniform-ish generator.

## Edge cases observed (synthetic data)

| Case | Handling |
|------|----------|
| Cancel/delete referencing an order_id never seen | counted in `stats.unmappedRefs`, no engine call |
| Execution drains an order to 0 | order removed from book, lobster_id unmapped |
| Halt (type 7) | logged, no engine state change |
| Hidden execution (type 5) | skipped (no displayed-book change in LOBSTER) |

Edge cases unique to real LOBSTER data that the synthetic generator does not exercise (Phase 2 work):
- Opening/closing auctions (large block prints at 09:30 / 16:00)
- Trading halts that span hundreds of events
- Cross trades (type 6) at top of book
- Replace-cross-side (rare in equities, more common in options)

## Constraints honored

- No modification to `engine.js`, `app.js`, `index.html`, or the existing `tests/test_invariants.html`. All cross-engine state mutation is done externally on the public `book.bids` / `book.asks` arrays per the layering note above.
- Pure Node stdlib (`fs`, `readline`, `assert`, `path`, `os`). Zero npm dependencies.
- Streaming parser. Both message and orderbook CSVs are read line-by-line and advanced in lockstep; neither is fully loaded into memory.
- LOBSTER data files are gitignored; only code and tests are tracked.
